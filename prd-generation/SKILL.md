# PRD Generation

## Core Rule

先澄清，再写 PRD。不要把残缺需求扩写成看似完整的文档。

产物是业务向中文 PRD，面向业务方、产品、设计、研发、测试共同评审。它描述业务背景、目标、用户、流程、系统交互、功能行为、非功能需求和未解决问题；不要写成 TRD，不展开 DB 主键、字段表、SQL、接口 schema、缓存结构等纯技术细节。

PRD 必须站在用户视角，并写清交互需求：要有用户旅程、用户故事、痛点，要为关键页面定义状态（默认/空/加载/成功/失败/无权限/异常）及对应文案。但同样有边界——只写交互需求与状态，不画线框、不定具体视觉/布局/控件样式，那是交互/视觉设计稿的职责。

### 语言与读者

默认读者是业务方——评审中最不懂技术的人。主线叙述必须让业务方不靠研发翻译就能读懂。

把两种语言分开：

- **主线叙述**（背景、目标、用户、场景、功能要解决什么、用户会看到什么）→ 平实、场景化、讲价值，不用实现术语。
- **必要的精确区块**（系统交互图、非功能需求里的性能/安全/埋点）→ 明确标注为“技术区块”，可用专业术语，与主线隔离。

工程术语能不用就不用；必须用时第一次出现用一句话解释。常见“工程话 → 人话”改写：

| 工程话 | 人话 |
| --- | --- |
| 接口超时/失败 | 系统繁忙、加载失败 |
| 鉴权/校验身份 | 确认是否登录或是否有权限 |
| 下发 | 发放/推送/展示 |
| 埋点 | 数据统计 |
| 并发/QPS | 访问量/高峰承载 |
| 幂等 | 防止重复提交 |
| 字段/参数 | 信息项 |
| token/会话 | 登录状态 |
| 降级/兜底（面向用户处） | 备用提示 |

### 价值优先

每个功能模块先讲“用户在什么场景、为了什么目标、对用户的价值”，再讲功能如何运作。不要上来就堆功能点和规则。

## Inputs

可接受的原始素材包括：

- 一句话产品想法
- 会议纪要、聊天记录、BRD 摘要
- 竞品说明、截图说明、用户反馈
- 已有 PRD 草稿，需要补全或重写

如果用户只说“帮我写 PRD/需求文档/产品方案”，也触发本 skill。

## Workflow

### 1. 吸收需求素材

先提取已知信息：

- 业务背景、问题与机会
- 目标用户、用户分群、使用场景
- 核心功能、入口、流程、异常分支
- 涉及系统、上下游依赖、上线节奏
- 目标指标、验收标准、风险约束

同时列出信息空白。不要脑补关键事实。

### 2. 澄清关键空白

写文档前必须先澄清。默认至少 1 轮；需求复杂、涉及多角色/多系统/风控合规时至少 2 轮。

如果当前宿主提供结构化提问工具，优先使用；否则直接在对话里用简体中文一次问 1-4 个问题，然后等待用户回复。

四类主题必须覆盖，已明确的信息可以用确认题快速带过：

| 主题 | 必须弄清楚的内容 |
| --- | --- |
| 业务背景与收益目标 | 为什么现在做、解决什么问题、目标指标、上线时间 |
| 功能模块与交互流程 | 主流程、入口、关键交互、异常分支、兜底文案 |
| 涉及系统与改造范围 | 前端、服务端、数据、第三方系统、是否需要配置后台 |
| 用户体验与交互 | 核心用户的目标/痛点、用户旅程、关键页面的空态/失败/无权限体验、反馈与二次确认预期 |
| 非功能需求 | 风控、客服、埋点、合规、财务/法务、性能容量、安全 |

提问要求：

- 每轮聚焦一个主题，避免一次抛 10 个问题。
- 尽量给 2-4 个可选方向，并允许用户补充“其他”。
- 涉及多用户分群时，单独追问分群规则和差异化链路。
- 涉及面向用户的页面时，必须追问关键状态（空态、失败、无权限）的体验与文案，不要默认只有正常路径。
- 用户暂时答不出来的内容，写入“未解决的问题”，不要硬填。

### 3. 使用模板组织 PRD

先读取 `references/template.md`，按模板生成 Markdown。章节顺序保持：

1. 文档修订记录
2. 概述
3. 总体流程
4. 功能需求
5. 非功能需求
6. 多期产品计划
7. 未解决的问题
8. 附录

章节可以写“无”或“不涉及”，但不要删除一级结构。新模块/新场景需要竞品分析；已有功能迭代可写“无直接竞品/不涉及”。

### 4. Mermaid 强制表达

能用图讲清楚的流程，不要写成长段文字。

必须使用 Mermaid 的场景：

| 场景 | 图类型 |
| --- | --- |
| 多用户分群、决策树 | `flowchart TB` |
| 业务主流程 | `flowchart TD` 或 `flowchart LR` |
| 页面流转（页面级跳转） | `flowchart LR`（区别于业务逻辑流程） |
| 多端/多系统交互 | `sequenceDiagram` |
| 产品架构/模块组成 | `graph TB` + `subgraph` |
| 复杂多状态页面 | `stateDiagram-v2` |
| 用户旅程/多角色协作 | `journey` 或泳道 `flowchart` |
| 时间节奏 | `gantt` 或表格，复杂时用 `gantt` |

判断标准：

- 一段流程超过 4 句话，画图。
- 出现 2 个以上分支判断，画图。
- 涉及 2 个以上系统调用，画时序图。
- 多触点/跨阶段的用户体验，画用户旅程图。

Mermaid 质量要求：

- 每张图前用 1-2 句中文说明“这张图在讲什么”。
- 节点文本保持短句，复杂逻辑拆成多张图。
- 业务流程图、用户旅程图、页面流转图的节点用用户/业务能懂的话（如“用户提交申请”，不是“调用提交接口”）；只有系统交互图使用前端/服务端/第三方等系统术语，并在图前标注“（技术视角，供研发/测试）”。
- 判断节点用 `{...}`，结束节点用 `([...])`。
- 异常、兜底、风控分支不能省略。
- 时序图参与方按“用户 → 前端 → 服务端 → 第三方/依赖系统”排列。

### 5. 写入文件

默认输出 Markdown 文件：

- 用户指定路径优先。
- 未指定时，写到当前项目 `prd-output/`。
- 文件名：`PRD_{产品或功能短名}_{YYYY-MM-DD}.md`。

使用当前环境提供的文件写入工具（如 Write/Edit）创建或更新 Markdown，遵守当前环境的文件编辑规则。

日期处理：把素材中的相对日期（“下周三”“本月底”“Q3”等）按当前环境提供的当天日期换算成绝对 `YYYY-MM-DD` 再写入；无法确定的日期写“待定”，不要留 `YYYY-MM-DD` 占位符。

### 6. 交付前自检

结束前逐条检查：

- 已至少进行 1 轮澄清，或用户明确要求基于现有材料直接生成并接受缺口。
- 文档为简体中文，技术术语可保留英文。
- 文档包含“文档修订记录 + 7 个一级章节”。
- 业务主流程至少 1 张 Mermaid。
- 涉及多端/多系统时，有系统交互 Mermaid。
- 涉及多用户分群时，有用户图谱 Mermaid。
- 已提取核心用户的目标/痛点，关键模块有用户故事或核心场景。
- 多触点/跨阶段体验时，有用户旅程 Mermaid。
- 关键面向用户的页面有状态体系（至少覆盖空/加载/成功/失败/异常）与对应文案。
- 非技术读者（业务方）能独立读懂主线，无需研发翻译。
- 每个功能模块先讲了用户价值与场景，再讲机制。
- 除明确标注的技术区块（系统交互图、非功能需求）外，正文没有裸用 接口/鉴权/QPS/埋点ID/缓存/降级/幂等 等工程术语（必要处已用人话解释或归入技术区块）。
- 没有 DB 主键、字段表、DDL、SQL、接口 schema 等 TRD 内容。
- 没有像素级布局、视觉规范、控件样式等设计稿内容。
- 日期均为绝对日期或明确“待定”，无残留 `YYYY-MM-DD` 占位符。
- 所有未确认内容集中写入“未解决的问题”。
- 最终回复给出文件绝对路径、未解决问题数量、Mermaid 图数量。

## Output Style

- 默认输出 1500-4500 字；材料少时宁可短而诚实，不要套话填满。
- 表格用 GitHub Flavored Markdown。
- 业务规则按“正常路径 / 判断规则 / 异常兜底 / 权限与边界”组织，用业务语言描述，不用实现术语。
- 交互规则按“状态 → 触发 → 反馈 → 文案”组织；用户视角用用户故事句式（作为…我想要…以便…）表达。
- 主线多用短句、场景化；优先用“用户看到 / 用户可以 / 如果…用户会…”句式，少用“系统校验 / 接口返回 / 服务端处理”。
- 文档里的“收益目标”和“成功指标”必须尽量量化；缺数据时写明待确认。

## Common Pitfalls

- 直接动笔：需求不完整时必须先问。
- 问题过多：每轮 1-4 个问题。
- 把 PRD 写成技术方案：PRD 讲业务规则和产品行为，不写实现细节。
- 流程无图：复杂流程必须 Mermaid。
- 只写功能不写用户体验：要有用户旅程、用户故事、痛点。
- 漏掉空态/异常态：关键页面要逐一定义状态及其文案，别只写正常路径。
- 把 PRD 写成视觉设计稿：只定交互需求与状态，不画线框、不抠布局视觉。
- 工程腔：把“加载失败 / 系统繁忙”写成“接口超时 / 降级”，业务方读不懂。
- 功能堆砌：只罗列功能点和规则，不讲用户为什么需要、在什么场景用、价值是什么。
- 语言不分层：把技术细节混进业务主线，而不是隔离到技术区块。
- 假装没有缺口：缺口写到“未解决的问题”。
- 模板填空味太重：每个章节都要围绕真实业务目标和使用场景写。
